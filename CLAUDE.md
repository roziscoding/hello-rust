# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## ⚠️ This is a hands-on Rust learning project — mentor, don't scaffold

The owner is learning Rust by building this themselves and writes **all** the code by hand. Their words: *"I wanna do it all by hand, otherwise I won't learn."*

- **Do NOT write, scaffold, or edit the source with Edit/Write tools** unless explicitly asked (e.g. "write the commit"). Default to teaching.
- Act as a guide: explain concepts, give precise tasks, point at the right APIs, and **review** what they write — use `Read` to inspect, and run `cargo build`/`test`/`clippy` to give grounded feedback.
- Add dependencies **just-in-time**, only when a step actually needs them — don't pre-add crates.
- They value understanding the *why* (ownership, traits, lifetimes) and often reach correct conclusions on their own. Let them.

## Code style preference

- **Explicit `return` everywhere**, including the final expression of a function. This is intentional; `clippy::needless_return` is set to `allow` in `Cargo.toml`. Do not suggest implicit/trailing-expression returns for function bodies. (Trailing expressions are still required inside `match` arms, `if` expressions, and closures, since `return` exits the whole function.)

## Commands

**Always use `mise` tasks. Never run `cargo`, `bacon`, or other tools directly.** Tasks are defined in `.mise/tasks/`.

| Goal | Command |
| --- | --- |
| Dev watch (auto-restart server) | `mise run dev` |
| Run the server once (no watch) | `mise run start` |
| Lint (clippy, pedantic) | `mise run dev:check` |
| Run tests | `mise run test` |
| Run a single test | `mise run test <name>` (e.g. `mise run test update_returns_none`) |
| Watch tests | `mise run test:watch` |

- The `test` and `start` tasks forward extra arguments to `cargo` (`"$@"`), so `mise run test <name>` filters by test name.
- Tooling (`rust`, `bacon`, `watchexec`) is pinned via `mise.toml`; `mise install` sets it up.
- If a workflow has no task yet, **add a `mise` task for it** (in `.mise/tasks/`) rather than reaching for a raw command.
- Read **full** compiler diagnostics — `mise run dev:check` prints the complete `help:` lines that VSCode truncates.

## Architecture

A small in-memory CRUD REST API for "friends" (`name`, `pronouns`, optional `notes`), built on **axum 0.8 / tokio**, with `serde` (JSON), `validator` (input rules), and `uuid` (v4 IDs). No database — state lives in process memory. Rust edition 2024.

Request flow and module responsibilities:

- **`main.rs`** — builds the `FriendRepository` (the app state), wires routes (`/friends`, `/friends/{id}` — note axum 0.8 uses `{id}`, not `:id`), and serves. The repository is injected into every handler via axum's `.with_state(...)` (dependency injection).
- **`models.rs`** — `Friend` (stored shape, includes server-generated `id`) and `NewFriend` (request body shape, no `id`, carries `#[validate(...)]` rules). The wire/input type is deliberately separate from the stored type.
- **`handlers.rs`** — thin HTTP layer. Each handler receives `State<FriendRepository>`, calls `friend_params.validate()?` where applicable, then delegates to the repository. Knows nothing about locks or `HashMap`.
- **`repository.rs`** — `FriendRepository` owns the store: `Arc<RwLock<HashMap<Uuid, Friend>>>` (type alias `Db`). All lock acquisition (`std::sync::RwLock`, via private `read()`/`write()` helpers) and `clone`-out-of-the-map logic is encapsulated here. Methods are **synchronous** and short — they never hold a guard across `.await` (use `std::sync`, not `tokio::sync`, deliberately). `seed(...)` is a `#[cfg(test)]`-only constructor for injecting state in tests.
- **`errors.rs`** — `AppError` enum is the single error type returned by fallible handlers. `impl IntoResponse for AppError` maps each variant to an HTTP status + JSON body (`404` for `NotFound`, `422` for `Validation`). `impl From<ValidationErrors>` is what lets `validate()?` auto-convert into `AppError` via `?`. `JsonError` is a private, consistent error envelope (`code` / `message` / optional `details`).

Key cross-cutting points:

- **Validation is run manually in handlers** (`.validate()?`), not via an extractor/middleware. axum's `Json<NewFriend>` only checks JSON *shape*; value rules come from the explicit `validate()` call.
- **The error seam:** because handlers return `Result<_, AppError>` and `From<ValidationErrors>`/`ok_or_else(AppError::not_found(...))` feed into it, every failure path funnels through `AppError::into_response` — add a new failure mode by adding an enum variant + a match arm.
- **The storage seam:** swapping the in-memory store for a real DB should only touch `repository.rs`; handlers/models/errors are insulated from it.
